Implementation Plan - Custom gRPC Queuing Latency Interceptor (Datastore Specific)
This plan outlines the implementation of a custom gRPC client interceptor to measure the latency between when a user initiates a request and when the request is actually sent on the wire. This is to help diagnose latency caused by exceeding MAX_CONCURRENT_STREAMS.

Based on user feedback, this will be implemented within the Datastore library for now and will ensure it works across all channels in a ChannelPool and flows properly through GAX.

Proposed Changes
Java Datastore
[NEW] 
GrpcQueuingLatencyInterceptor.java
Create a new interceptor that:

Captures the start time when a call is initiated (by wrapping ClientCall.start).
Uses CallOptions.withStreamTracerFactory to register a ClientStreamTracer.
In ClientStreamTracer.streamCreated, captures the time when the stream is actually created on the transport.
Calculates the duration and adds it as an event or attribute to the current OpenTelemetry span.
[MODIFY] 
EnabledTraceUtil.java
Modify OpenTelemetryGrpcChannelConfigurator to add the new GrpcQueuingLatencyInterceptor to the channel builder.

java
public class OpenTelemetryGrpcChannelConfigurator
        implements ApiFunction<ManagedChannelBuilder, ManagedChannelBuilder> {
  
      @Override
      public ManagedChannelBuilder apply(ManagedChannelBuilder managedChannelBuilder) {
        GrpcTelemetry grpcTelemetry = GrpcTelemetry.create(getOpenTelemetry());
        return managedChannelBuilder.intercept(
            grpcTelemetry.newClientInterceptor(),
            new GrpcQueuingLatencyInterceptor() // Add custom interceptor here
        );
      }
    }
Flow Analysis (Datastore -> GAX -> gRPC-Java)
The user initiates a call via Datastore client -> GAX Callables (GrpcDirectCallable, etc.) -> gRPC Channel (ManagedChannel). Since we are registering the interceptor on the ManagedChannel level (via channelConfigurator in EnabledTraceUtil which is passed to InstantiatingGrpcChannelProvider), it will intercept all calls that flow through the channel, regardless of whether they were initiated by direct stubs or by GAX callables.

This ensures that the queuing latency measurement will capture delays occurring just before the request is sent to the wire by gRPC-Java/Netty, after all GAX-level logic has been executed.

ChannelPool Support
By adding the interceptor via ApiFunction<ManagedChannelBuilder, ManagedChannelBuilder> in EnabledTraceUtil, and passing it to InstantiatingGrpcChannelProvider (via DatastoreOptions), it will be applied to all channels created by the ChannelPool. This is because ChannelPool uses the provider's createSingleChannel method (which calls this configurator) for each channel it instantiates.
